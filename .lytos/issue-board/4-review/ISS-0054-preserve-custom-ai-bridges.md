# ISS-0054 — Preserve custom AI bridge files on init/upgrade instead of overwriting them

## Context

Advanced teams enrich generated bridge files such as `CLAUDE.md`, `AGENTS.md`, `.github/copilot-instructions.md`, or `.cursor/rules/lytos.mdc` with project-specific instructions. Today `lyt init --force` and follow-up scaffold flows can overwrite those files bluntly.

That is a real adoption risk. The generic bridge generated by Lytos is useful, but once a team has added stack- or project-specific guidance, overwriting it destroys value and creates distrust around updates.

The desired behavior is not "never touch bridges again". It is "preserve user-authored content by default, and make destructive overwrite an explicit choice".

## Proposed solution

1. Distinguish between:
   - generated canonical bridge content owned by Lytos
   - user customizations layered on top
2. On `lyt init --force`, do not overwrite an existing bridge file silently.
3. Introduce an explicit strategy for bridge conflicts:
   - default: keep existing file and warn
   - optional explicit overwrite flag for full replacement
   - follow-up design option: merge/patch markers if a safe merge strategy is feasible
4. Apply the same protection logic to any workflow that refreshes or regenerates bridge files.
5. Document the behavior clearly in CLI help and docs so teams know what is preserved and what is not.

## Notes

- This issue comes directly from external AI-user feedback on a real project.
- The product expectation is "Lytos can be adopted safely in a customized repo", not "Lytos always owns the full bridge file forever".

## Implementation notes

- Implemented in `init/scaffold` with a preserve-by-default bridge write path.
- New explicit escape hatch: `--overwrite-bridges`.
- Covered by integration tests for:
  - root bridge preservation (`CLAUDE.md`)
  - explicit overwrite with `--overwrite-bridges`
  - nested bridge preservation (`.github/copilot-instructions.md`)
- Full test suite green at review handoff: `112/112`.
